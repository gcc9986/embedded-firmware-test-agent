# 申请材料可直接使用版本

## 项目名称

嵌入式研发测试闭环 Agent：固件测试与故障定位自动化系统

## 项目解决的核心痛点

我构建了一个面向嵌入式研发流程的自动化测试与故障定位 Agent，主要解决固件开发中“编译、烧录、串口日志分析、问题同步”高度依赖人工的问题。过去每次版本提交后，工程师需要手动拉取代码、编译固件、烧录开发板、观察串口输出，并把异常整理到飞书群或任务表中，单次流程通常需要 20-40 分钟，且容易遗漏关键日志。

该系统接入 Git、CI、串口调试工具、飞书机器人和测试脚本，在研发提交代码后自动完成固件编译、烧录、串口日志采集、异常归因、测试报告生成，并把问题同步到飞书群或任务系统中。它重点解决以下问题：

1. 固件测试链路长，反馈慢。
2. 编译日志、串口日志、CI 日志和历史问题分散，定位成本高。
3. 飞书群里的问题描述不标准，缺少复现步骤、关键证据和修复建议。
4. 常见配置类、编译类、外设通信类问题依赖资深工程师经验，难以沉淀。
5. 硬件偶发问题与代码问题容易混淆，需要自动化证据链辅助判断。

## 核心逻辑流

核心流程是：首先监听代码提交或飞书指令，自动判断测试目标；然后调用编译工具链生成固件；接着通过脚本完成设备烧录和串口日志采集；随后由日志分析模块对编译错误、运行日志和历史问题库进行分析，判断是代码逻辑错误、配置错误、外设通信异常还是偶发硬件问题；最后自动生成测试结论、复现步骤、可能原因和修复建议，并同步到飞书群。

系统包含长链推理和多 Agent 协作。长链推理体现在它不是只看一段日志，而是按“提交 diff → 编译日志 → 烧录结果 → 串口日志 → 历史问题库 → 测试结论”的证据链逐步判断。多 Agent 协作结构包括：调度 Agent 负责任务拆解，代码分析 Agent 负责理解提交 diff，构建 Agent 负责编译固件，烧录 Agent 负责下载固件，串口 Agent 负责运行日志采集，日志分析 Agent 负责异常归因，报告 Agent 负责生成飞书消息和测试记录。

实际使用后，单次测试反馈时间可以从约 30 分钟降低到 6-8 分钟，人工整理日志时间减少约 70%，常见编译和配置类问题的定位准确率约 85%。这个项目让我真正把 AI 从“问答工具”变成了能调用真实研发工具链、完成可验证工作的自动化 Agent。
